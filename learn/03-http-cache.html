#3 HTTPヘッダーとキャッシュ設計

影響指標: FCP, LCP | 難易度: 基礎

1. なぜキャッシュが重要か

ブラウザがリソースをリクエストするたびにサーバーまで往復すると、ネットワーク遅延（RTT）分だけ表示が遅れます。キャッシュを使えば2回目以降のアクセスでこの往復を省略でき、FCP/LCPが大幅に改善します。

キャッシュなし: Browser ──GET /app.js──▶ Server ──200 + body──▶ Browser (100ms RTT + 50ms処理 = 150ms) キャッシュあり (immutable): Browser ──(ローカルキャッシュから取得)──▶ 即座に使用 (0ms) キャッシュあり (ETag): Browser ──If-None-Match: "abc"──▶ Server ──304 Not Modified──▶ Browser (100ms RTT、bodyなし = 100ms)

2. Cache-Control ヘッダーの主要ディレクティブ

ディレクティブ意味使いどころ
no-storeキャッシュ禁止。一切保存しない個人情報を含むAPIレスポンス
no-cacheキャッシュするが、毎回サーバーに再検証するHTML（最新を返したいが、304は使いたい）
max-age=NN秒間はキャッシュを新鮮とみなすAPI、短期キャッシュ
immutableリソースが変わらないことを宣言（再検証もしない）ハッシュ付きファイル名の静的アセット
publicCDNなど共有キャッシュにも保存OK全ユーザー共通のリソース

3. WSH 2026 で何をしたか

課題アプリには2つの問題がありました：

Before: 意図的にキャッシュを殺していた
// app.ts - 全レスポンスにキャッシュ無効ヘッダーを付与
app.use((_req, res, next) => {
  res.header({
    "Cache-Control": "max-age=0, no-transform",
    "Connection": "close",
  });
  return next();
});

// static.ts - ETag/LastModified も無効化
serveStatic(PUBLIC_PATH, {
  etag: false,
  lastModified: false,
});
After: 適切なキャッシュ戦略に変更
// app.ts - グローバルミドルウェアを削除（各ルートで個別設定）

// static.ts - 静的ファイルに長期キャッシュ
serveStatic(UPLOAD_PATH, { maxAge: "1d" });   // ユーザーアップロード
serveStatic(PUBLIC_PATH, { maxAge: "7d" });   // 公開アセット
serveStatic(CLIENT_DIST_PATH, { maxAge: "7d" }); // ビルド出力
// etag, lastModified はデフォルト true に戻る

4. キャッシュ設計のベストプラクティス

リソース種別ごとのキャッシュ戦略: HTML: Cache-Control: no-cache → 毎回サーバー確認、ETag で 304 を狙う ハッシュ付きJS/CSS: Cache-Control: public, max-age=31536000, immutable (app.a1b2c3.js) → 1年間キャッシュ、ファイル名が変われば別リソース API: Cache-Control: no-cache or max-age=60 → ケースバイケース 画像/フォント: Cache-Control: public, max-age=86400 → 1日キャッシュ（変更頻度による）

ポイント: immutable + ハッシュ付きファイル名

Rspack/Webpack は出力ファイル名にコンテンツハッシュを含められます（例: app.a1b2c3.js）。コードが変われば新しいハッシュが付くので、古いキャッシュが残っていても問題ありません。この組み合わせが最強のキャッシュ戦略です。

5. ETag と Last-Modified

ETag はリソースの「指紋」（ハッシュ値）、Last-Modified は最終更新日時です。ブラウザは次回リクエスト時にこれらを送り、サーバーは変更がなければ 304 Not Modified（ボディなし）を返します。

WSH の罠: etag: false, lastModified: false

Express の serveStatic でこれらを false にすると、304 応答が使えなくなり、毎回フルボディを返すことになります。WSH では意図的にこれが設定されていました。

6. Connection: close の問題

Connection: close は HTTP/1.1 のKeep-Alive（接続の再利用）を無効にします。リソースごとに新規TCP接続が必要になり、TLSハンドシェイクのコストが毎回かかります。HTTP/2 では多重化されるため影響は少ないですが、HTTP/1.1 環境では致命的です。

確認クイズ: max-age=0 と no-cache の違いは？

max-age=0: キャッシュの有効期限が0秒。ブラウザはキャッシュを持つが、毎回期限切れとして再検証する。

no-cache: キャッシュを保存するが、使用前に必ずサーバーに再検証する。

実質的にはほぼ同じ動作ですが、no-cache の方が意図が明確です。max-age=0, no-transform のように複数ディレクティブを組み合わせると、意図しない挙動になることがあります。

確認クイズ: immutable を付けるべきでないファイルは？

HTMLファイル。HTMLにimmutableを付けると、HTMLが更新されてもブラウザは古いHTMLを使い続け、新しいJSやCSSへの参照が更新されません。HTMLは必ず no-cache にし、JSやCSSのファイル名にハッシュを含めてimmutableにするのが正解です。
