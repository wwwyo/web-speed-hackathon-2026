#4 レスポンス圧縮

影響指標: FCP, LCP (転送量削減) | 難易度: 基礎

1. HTTPレスポンス圧縮とは

サーバーがレスポンスボディを gzip や Brotli で圧縮して送り、ブラウザが展開する仕組みです。テキストベースのリソース（HTML, JS, CSS, JSON）は70-90%のサイズ削減が可能です。

圧縮なし: Browser ──GET /app.js──▶ Server Browser ◀──200 (500KB body)── Server 転送量: 500KB gzip 圧縮: Browser ──GET /app.js──▶ Server Accept-Encoding: gzip Browser ◀──200 (120KB body)── Server Content-Encoding: gzip 転送量: 120KB (76%削減!)

2. 圧縮のネゴシエーション

ブラウザとサーバーは以下のように圧縮方式を合意します:

  1. ブラウザが Accept-Encoding: gzip, deflate, br ヘッダーを送る
  2. サーバーが対応する圧縮方式でボディを圧縮
  3. サーバーが Content-Encoding: gzip ヘッダーを付けて返す
  4. ブラウザが自動的に展開して使用

3. 圧縮アルゴリズムの比較

アルゴリズム圧縮率速度ブラウザ対応備考
gzip良い速い全ブラウザ最も無難な選択
Brotli (br)最良 (gzip比 15-25%改善)圧縮は遅いChrome, Firefox, Safari, EdgeHTTPS必須
zstd良い最速Chrome 123+まだ普及途上

4. Express での実装

WSH 2026 では compression ミドルウェアを1行追加しただけです:

実装（たった3行の変更）
import compression from "compression";

const app = Express();
app.use(compression());  // ← これだけ！
app.use(sessionMiddleware);
// ...以降のミドルウェア

compression ミドルウェアの内部動作

  1. Accept-Encoding ヘッダーを確認
  2. レスポンスの Content-Type が圧縮対象か判定（テキスト系のみ、画像・動画は除外）
  3. ボディサイズが閾値（デフォルト1KB）以上なら圧縮
  4. Node.js の zlib モジュールで gzip 圧縮を実行
  5. Content-Encoding: gzip と Vary: Accept-Encoding を付与

5. 圧縮すべきもの/すべきでないもの

圧縮すべき圧縮不要
HTML, CSS, JavaScriptJPEG, PNG, WebP (既に圧縮済み)
JSON, XML, SVGMP4, WebM (既に圧縮済み)
フォント (WOFF2以外)WOFF2 (既にBrotli圧縮済み)
ソースマップ小さいファイル (1KB未満)
注意: 圧縮のCPUコスト

動的圧縮はリクエストのたびにCPUを使います。静的ファイルは事前圧縮（ビルド時に .gz/.br ファイルを生成）する方が効率的です。WSH のような短時間競技では動的圧縮で十分ですが、本番サービスでは CDN + 事前圧縮が理想です。

6. 事前圧縮のアプローチ

# ビルド時に Brotli と gzip の事前圧縮ファイルを生成
find dist -type f \( -name "*.js" -o -name "*.css" -o -name "*.html" \) \
  -exec brotli {} \; -exec gzip -k {} \;

# → dist/app.js, dist/app.js.br, dist/app.js.gz が生成される

Express の express-static-gzip ミドルウェアを使うと、ブラウザの Accept-Encoding に応じて .br → .gz → 元ファイルの順に返せます。

確認クイズ: Vary: Accept-Encoding はなぜ必要？

CDN やプロキシキャッシュが、同じURLでも Accept-Encoding が異なるリクエストを区別してキャッシュするためです。これがないと、gzip対応ブラウザにキャッシュされた非圧縮レスポンスが返されたり、その逆が起きます。

確認クイズ: WOFF2 フォントを gzip 圧縮するとどうなる？

WOFF2 は内部で Brotli 圧縮済みなので、さらに gzip してもほとんどサイズが変わらない（むしろ微増することも）。二重圧縮のCPUコストだけが無駄になります。compression ミドルウェアはデフォルトで Content-Type をチェックし、application/font-woff2 は圧縮対象外にしています。
