#5 HTML最適化とレンダリング開始タイミング

影響指標: FCP, LCP, TBT | 難易度: 基礎

1. script タグの読み込み方式

ブラウザが HTML を上から順にパース（解析）するとき、<script> タグの書き方によって挙動が大きく変わります。

■ 属性なし（デフォルト） <script src="main.js"></script> HTML: ████████──STOP──████████████████████▶ JS: ▼download▼▼execute▼ ↑ HTMLパースが完全に止まる（parser blocking） ■ async <script async src="main.js"></script> HTML: ██████████████████──STOP──███████████▶ JS: ▼download▼ ▼execute▼ ↑ DL中はパース続行、実行時だけ止まる ※ 実行順序は保証されない ■ defer <script defer src="main.js"></script> HTML: ████████████████████████████████████▶ JS: ▼download▼ ▼execute▼ ↑ DL中もパース続行 ↑ HTMLパース完了後に実行 ※ 記述順に実行される
属性HTMLパースをブロック実行タイミング実行順序
なしする（最悪）DL完了直後記述順
asyncDL中はしない、実行中はするDL完了直後保証なし
deferしない（最良）HTMLパース完了後記述順

2. WSH 2026 で何が起きていたか

Before: 二重のブロッキング
<!-- index.html -->
<script src="/scripts/main.js"></script>  <!-- ← 属性なし = parser blocking -->
// index.tsx
window.addEventListener("load", () => {    // ← load イベントまで待つ
  createRoot(document.getElementById("app")!).render(
    <Provider store={store}>
      <BrowserRouter>
        <AppContainer />
      </BrowserRouter>
    </Provider>,
  );
});

問題点は2つ：

問題1: script タグに属性がない

HTML パースの流れ: <head> <meta ...> <title>CaX</title> <script src="/scripts/main.js"></script> ← ここでパース停止！ <link rel="stylesheet" ...> ← JSのDL+実行が終わるまで ... 読み込まれない </head> <body> <div id="app"></div> ← body もまだ存在しない </body> 結果: JS のダウンロード+実行中、ページは真っ白

問題2: load イベントを待っている

load イベントのタイミング: HTML パース → DOM 構築 → CSS/画像/フォント 全部DL → load 発火 ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━▶ ↑ ここでやっと React 起動 全リソース待ちは不要！ DOMContentLoaded のタイミング: HTML パース → DOM 構築 → DOMContentLoaded 発火 ━━━━━━━━━━━━━━━━━━━━━▶ ↑ defer スクリプトはここで実行される 画像やフォントのDLを待たない

load イベントは画像・フォント・CSS 等の全リソースのダウンロード完了後に発火します。React の初期化に全リソース完了を待つ必要は全くありません。

3. 修正内容

After: defer + 即座レンダリング
<!-- index.html -->
<script defer src="/scripts/main.js"></script>  <!-- defer 追加 -->
// index.tsx - load イベントのラッパーを削除
createRoot(document.getElementById("app")!).render(
  <Provider store={store}>
    <BrowserRouter>
      <AppContainer />
    </BrowserRouter>
  </Provider>,
);

改善効果の流れ:

Before: ──[HTML parse]──STOP──[JS DL]──[JS exec]──[HTML]──[CSS DL]──[load]──[React]──▶ ↑ ここまで真っ白 After: ──[HTML parse ═══════════════════════════]──[JS exec]──[React]──▶ ↑ JS は並行DL（パース止めない） ↑ パース完了直後に実行 FCP/LCP の改善: 数百ms 〜 数秒の短縮

4. defer vs async：どちらを使うべきか

場面推奨理由
アプリのメインJSdeferDOM が必要 + 実行順序を保証したい
アナリティクス (GA等)asyncDOM 不要 + 他のJSと順序依存なし
インラインスクリプトどちらも効かない外部ファイルにのみ有効
type="module"不要（デフォルトで defer 相当）ES Modules は仕様で defer 動作
注意: defer は DOM 存在を保証する

defer スクリプトは HTML パース完了後（DOMContentLoaded 直前）に実行されます。つまり document.getElementById("app") は必ず成功します。だからこそ load イベントを待つ必要がないのです。

5. ブラウザのリソース読み込みパイプライン

ブラウザがページを表示するまで: 1. HTML をダウンロード 2. HTML をパース（DOM ツリー構築） ├─ <link rel="stylesheet"> → CSS ダウンロード → CSSOM 構築 ├─ <script> (defer なし) → ★パース停止★ → JS DL+実行 ├─ <script defer> → バックグラウンドで JS DL（パース継続） └─ <img>, <font> → バックグラウンドで DL 3. DOM + CSSOM → Render Tree 構築 4. Layout（レイアウト計算） 5. Paint（描画）→ FCP ★ 6. defer スクリプト実行 → React レンダリング → LCP ★ 7. 画像・フォント等 全DL完了 → load イベント

defer を使うことで、ステップ2の「パース停止」がなくなり、FCP が大幅に早まります。さらに load を待たないことで、ステップ6→7の待ち時間もなくなり、LCP も改善します。

6. さらに進んだ最適化

  • preload: <link rel="preload" href="main.js" as="script"> でブラウザの preload scanner より早くDL開始
  • modulepreload: ES Modules の場合は <link rel="modulepreload"> で依存チェーンを事前解決
  • Critical CSS のインライン化: render-blocking な外部CSSを減らし、初期表示に必要な CSS だけ <style> に埋める
確認クイズ: DOMContentLoaded と load の違いは？

DOMContentLoaded: HTML のパースが完了し、DOM ツリーが構築された時点で発火。CSS、画像、iframe のダウンロード完了は待たない。defer スクリプトの実行完了後に発火。

load: ページの全リソース（画像、CSS、フォント、iframe 等）のダウンロードが完了した時点で発火。

React SPA の場合、DOM が存在すれば十分なので DOMContentLoaded 相当のタイミング（= defer スクリプト実行時）で起動するのが最適です。

確認クイズ: <script> が <head> にあるのと <body> 末尾にあるのは何が違う？

<head> に属性なし: HTML パースの冒頭でブロック。最悪の配置。

<body> 末尾に属性なし: HTML パースがほぼ終わった後にブロック。昔のベストプラクティスだった。

<head> に defer: DL は早い段階で開始（preload scanner が検出）しつつ、パースをブロックしない。現在のベストプラクティス。body末尾より DL 開始が早い分だけ有利。

確認クイズ: async と defer を両方付けたらどうなる？

仕様上は async が優先されます。ただし、async をサポートしない古いブラウザでは defer にフォールバックします。現代のブラウザでは両方付ける意味はなく、意図に応じてどちらか一方を使うべきです。
