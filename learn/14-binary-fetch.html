#14 バイナリ取得の廃止

影響指標: LCP, TBT | 難易度: 基礎

1. 問題: 画像表示に4ステップもかかっていた

Before の処理フロー: 1. fetchBinary(url) → ArrayBuffer をダウンロード 2. sizeOf(Buffer.from(data)) → image-size で幅/高さを取得（JS実行） 3. URL.createObjectURL(blob) → Blob URL を生成 4. <img src={blobUrl}> に設定 5. JS で containerRatio vs imageRatio を計算 6. absolute + translate で手動 cover After の処理フロー: 1. <img src={url}> → ブラウザが直接読み込み（以上！）

2. Before のコードが遅い理由

Before: 全てJSで制御
const { data, isLoading } = useFetch(src, fetchBinary);

// JS でサイズ計算
const imageSize = useMemo(() => {
  return data != null
    ? sizeOf(Buffer.from(data))  // ← image-size + Buffer polyfill
    : { height: 0, width: 0 };
}, [data]);

// Blob URL 生成
const blobUrl = useMemo(() => {
  return data != null
    ? URL.createObjectURL(new Blob([data]))
    : null;
}, [data]);

// コンテナサイズ取得
const [containerSize, setContainerSize] = useState({ height: 0, width: 0 });
const callbackRef = useCallback((el) => {
  setContainerSize({
    height: el?.clientHeight ?? 0,
    width: el?.clientWidth ?? 0,
  });
}, []);

// 手動 cover 計算
const containerRatio = containerSize.height / containerSize.width;
const imageRatio = imageSize.height / imageSize.width;

<img
  className={containerRatio > imageRatio ? "w-auto h-full" : "w-full h-auto"}
  style={{ position: "absolute", transform: "translate(-50%, -50%)" }}
  src={blobUrl}
/>

問題点の内訳:

処理問題コスト
fetchBinary画像をJSのArrayBufferとして取得。ブラウザの画像デコードパイプラインを使えないメモリ2倍（ArrayBuffer + img）
image-size + Buffer画像ヘッダーをパースしてサイズ取得。Buffer polyfill（~50KB）が必要TBT増加
URL.createObjectURLBlob URL の生成と管理。メモリリーク防止に revokeObjectURL が必要メモリリーク
手動 cover 計算JSでアスペクト比を計算し、条件分岐でCSS classを切替えリサイズで再計算

3. After: CSS object-fit に任せる

After: 2行で完了
<div className="relative h-full w-full overflow-hidden">
  <img
    alt={alt}
    className="h-full w-full object-cover"
    src={src}
  />
</div>

object-fit の値

値動作使いどころ
coverコンテナを埋める。はみ出す部分はクリップサムネイル、バナー画像
contain全体が見えるように収める。余白ができるロゴ、アイコン
fillコンテナにぴったり伸縮（アスペクト比無視）ほぼ使わない
none元サイズのままピクセルパーフェクトな表示
scale-downnone と contain の小さい方拡大は防ぎたい場合

4. なぜブラウザに任せるべきか

  • Progressive decoding: ブラウザは画像を段階的にデコードできる。fetchBinary ではダウンロード完了まで何も表示されない
  • キャッシュの効率: <img src> はブラウザのHTTPキャッシュ、メモリキャッシュ、ディスクキャッシュを自然に活用
  • Lazy loading: loading="lazy" が使える（fetchBinary では自前実装が必要）
  • Priority hints: fetchpriority="high" が使える
  • メモリ効率: JS heap にバイナリを保持しないので GC 負荷が減る
確認クイズ: object-fit: cover の等価な CSS は？

Before のコードが手動実装していたのと同じロジック:

position: absolute;
top: 50%;
left: 50%;
transform: translate(-50%, -50%);
/* + JS でアスペクト比を比較して width:auto/height:100% or width:100%/height:auto を切替え */

object-fit: cover はこの全てをブラウザの内部実装で行います。JS は一切不要で、GPU アクセラレーションも効きます。
