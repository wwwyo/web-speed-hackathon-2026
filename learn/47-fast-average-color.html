#47 メインスレッドブロック

影響指標: TBT, INP | 難易度: 中級

1. Long Task とは

ブラウザのメインスレッドで50ms以上かかるタスクを Long Task と呼びます。Long Task の間、ユーザーのクリックやキー入力への応答が遅れます。

メインスレッドのタイムライン: 良い状態: ──[5ms]──[3ms]──[10ms]──[2ms]──[8ms]──▶ ↑ ユーザー操作に即座に応答可能 悪い状態 (Long Task): ──[████████████ 200ms ████████████]──[3ms]──▶ ↑ この200msの間、一切の操作が無視される Lighthouse はこの超過分(200-50=150ms)をTBTに加算 TBT = Sum of (Long Task duration - 50ms)

2. FastAverageColor の問題

FastAverageColor はcanvasに画像を描画し、ピクセルデータを走査して平均色を計算するライブラリです。

Before: mode: "precision"
const fac = new FastAverageColor();
const { rgb } = fac.getColor(ev.currentTarget, { mode: "precision" });
// ↑ 全ピクセルを走査（1000x1000px = 100万ピクセル × 4チャンネル）
// メインスレッドを数十〜数百msブロック

precision モードの内部動作

precision mode: 1. canvas に画像を描画 2. getImageData() で全ピクセルの RGBA を取得 3. 全ピクセルを走査して R,G,B の平均を計算 for (let i = 0; i < data.length; i += 4) { sumR += data[i]; // Red sumG += data[i+1]; // Green sumB += data[i+2]; // Blue } → 1000x1000px の画像: 4,000,000 回のループ → メインスレッドで数十ms ブロック
After: mode: "speed"
const fac = new FastAverageColor();
// Use the fast sampling mode to avoid blocking the main thread on large images.
const { rgb } = fac.getColor(ev.currentTarget, { mode: "speed" });
// ↑ 画像を小さな canvas にリサイズしてからサンプリング
// 数ms で完了

speed モードの内部動作

speed mode: 1. 小さな canvas (10x10 程度) に画像を縮小描画 → ブラウザが自動的にリサイズ（ハードウェアアクセラレーション） 2. 100ピクセルだけ走査して平均色を計算 → ほぼ瞬時（<1ms） → 精度は若干落ちるが、背景色の決定には十分

3. Long Task を見つける方法

DevTools Performance パネル

  1. Chrome DevTools → Performance タブ
  2. Record & Reload（ページ読み込みを記録）
  3. Main スレッドの赤い三角マーク = Long Task
  4. クリックすると、どの関数が何ms使ったか確認できる

Performance Observer API

// コード内で Long Task を検出
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    console.warn(`Long Task detected: ${entry.duration}ms`);
  }
});
observer.observe({ type: "longtask", buffered: true });

パターン手法例

確認クイズ: TBT と INP の違いは？

TBT (Total Blocking Time): ページ読込中の Long Task の超過時間の合計。FCP〜TTI の間が対象。Lighthouse lab メトリクス。

INP (Interaction to Next Paint): ユーザーの実際の操作（クリック、タップ、キー入力）から画面更新までの時間のうち最も遅いもの。CrUX フィールドメトリクス。

TBT は「読込中にどれだけブロックされたか」、INP は「操作時にどれだけ待たされたか」。両方改善が必要です。
