#53 SSE + rAF によるTBT改善

影響指標: TBT, INP | 難易度: 上級

1. 問題の全体像

CaX の Crok AI チャットは SSE (Server-Sent Events) でストリーミング応答を受け取ります。SSE の各メッセージで React の state を更新し、Markdown をパースして再描画していました。

問題: SSEメッセージが高頻度で来ると... SSE: ─msg─msg─msg─msg─msg─msg─msg─msg─msg─msg─▶ (数十ms間隔) ↓ ↓ ↓ ↓ ↓ ↓ ↓ ↓ React: setState→render→setState→render→setState→render... ↓ ↓ ↓ DOM: update update update ← 毎回 Markdown パース + DOM更新 ████████████████████████████ = Long Task (TBT増加) 理想: 画面更新は 16ms (60fps) に1回で十分

2. 修正1: requestAnimationFrame バッチング (SSE)

Before: メッセージごとに setState
eventSource.onmessage = (event) => {
  const data = JSON.parse(event.data);
  const newContent = options.onMessage(data, contentRef.current);
  contentRef.current = newContent;
  setContent(newContent);  // ← 毎メッセージで React 再描画
};
After: rAF でバッチ化
eventSource.onmessage = (event) => {
  const data = JSON.parse(event.data);
  const newContent = options.onMessage(data, contentRef.current);
  contentRef.current = newContent;  // ref に最新値を保存

  // rAF でバッチ化: 16ms間に複数メッセージが来ても描画は1回
  cancelAnimationFrame(rafIdRef.current);
  rafIdRef.current = requestAnimationFrame(() => {
    setContent(contentRef.current);
  });
};

なぜこれが効くのか

Before: SSE: ─msg1──msg2──msg3──msg4──────────────msg5──▶ ↓ ↓ ↓ ↓ ↓ React: [render][render][render][render] [render] ↑ 4回レンダリング（16ms以内に4回は無駄） After (rAF バッチング): SSE: ─msg1──msg2──msg3──msg4──────────────msg5──▶ ↓cancel ↓cancel ↓cancel ↓ ↓ rAF: [render] [render] ↑ 1回だけレンダリング（最新値を使う）

requestAnimationFrame は次の画面描画タイミング（約16ms後）にコールバックを予約します。cancelAnimationFrame で前の予約をキャンセルすれば、16ms間に複数メッセージが来ても描画は1回で済みます。

3. 修正2: ストリーミング中は Markdown パースをスキップ

Before: 毎回 Markdown → React 変換
<Suspense fallback={<p>{content}</p>}>
  <ChatMessageMarkdown content={content} />
  {/* ← ストリーミング中も毎回 react-markdown + rehype-katex を実行 */}
</Suspense>
After: ストリーミング中はプレーンテキスト
{isStreaming ? (
  // ストリーミング中: 軽い pre で表示（Markdown パースなし）
  <p className="whitespace-pre-wrap">{content}</p>
) : (
  // 完了後: リッチな Markdown 表示
  <Suspense fallback={<p>{content}</p>}>
    <ChatMessageMarkdown content={content} />
  </Suspense>
)}

4. 修正3: MutationObserver のスコープ限定 (DM)

Before: document.body 全体を監視
const observer = new MutationObserver(() => {
  scrollToBottom();  // スクロール位置を最下部に
});
observer.observe(document.body, {
  childList: true,
  subtree: true,
  characterData: true,  // ← body 配下の全テキスト変更で発火
});
After: メッセージリストだけ監視 + rAF
const el = messageListRef.current;

let rafId = 0;
const observer = new MutationObserver(() => {
  cancelAnimationFrame(rafId);
  rafId = requestAnimationFrame(scrollToBottom);
});
observer.observe(el, {
  childList: true,     // 子要素の追加/削除のみ
  subtree: true,
  // characterData は不要（テキスト変更を監視しない）
});

なぜ body 全体の監視が危険か

  • document.body に subtree: true + characterData: true = ページ上のあらゆるテキスト変更で発火
  • React の仮想DOM更新、他コンポーネントの state 変更、ツールチップ表示...全てがトリガーになる
  • 各発火で scrollToBottom() が即座に実行され、DOM レイアウト強制（forced reflow）が発生

5. まとめ: 高頻度イベントの3つの対策パターン

パターン手法適用場面
rAF バッチングcancelAnimationFrame + requestAnimationFrameSSE、WebSocket、scroll、resize
描画の簡略化ストリーミング中は軽い表示、完了後にリッチ表示チャット、ログビューア
監視スコープの限定MutationObserver の対象を必要最小限に自動スクロール、DOM変更検知
確認クイズ: debounce と rAF バッチングの違いは？

debounce: 最後のイベントから N ms 経過後に実行。「入力が止まったら」のパターン。setTimeout ベース。

rAF バッチング: 次の画面描画タイミングで実行。「表示できるタイミングで最新値を」のパターン。ブラウザの描画サイクルに同期。

SSE ストリーミングでは rAF の方が適切です。debounce だと最後のメッセージから Nms 待つ間に画面が固まって見え、rAF なら16ms(60fps)ごとに確実に更新されます。
