#64 ReDoS 正規表現の修正

影響指標: Security, TBT | 難易度: 中級

1. ReDoS とは

ReDoS (Regular Expression Denial of Service) は、特定の正規表現に対して悪意のある入力を与えることで、マッチング処理が指数関数的に遅くなる脆弱性です。

正常な入力: "abc!def" → マッチ判定: 0.01ms 悪意ある入力: "a".repeat(30) → マッチ判定: 数分〜数時間 !! なぜ？→ Catastrophic Backtracking（破滅的バックトラッキング）

2. Catastrophic Backtracking の仕組み

WSH 2026 で実際に存在した危険な正規表現を例に解説します:

危険な正規表現（パスワードチェック）
/^(?:[^\P{Letter}&&\P{Number}]*){16,}$/v

この正規表現は「文字または数字のみ16文字以上」にマッチします。問題は ([...]*)が{16,}回 のネスト量指定子です。

入力: "superultra_hyper_miracle_romantic" (記号 _ を含む32文字) 正規表現エンジンの動き: 1. まず * で "superultra" (10文字) を消費 → _ でマッチ失敗 2. バックトラック: "superultr" (9文字) にして、 次の繰り返しで "a" を試す... 3. さらにバックトラック: "superult" (8文字)... 組み合わせ数: 各文字を「どの繰り返しに割り当てるか」の 全パターンを試す → O(2^n) で爆発！ n=32 の場合: 2^32 = 約43億パターン → 数時間かかる

3. 危険パターンの見分け方

パターン例なぜ危険か
ネストした量指定子(a+)+, (a*)*, (a+){N}各文字の割当先が指数的に増える
重複する選択肢(a|a)+, (\d|\d)+どちらの選択肢も同じ文字にマッチ
重複する文字クラス + 量指定子[a-zA-Z]*[a-z]文字クラスの範囲が重なる
WSH 2026 の例((\d|\d{4}-\d{2}-\d{2})+)+$数字が両方にマッチし、$ で失敗時にバックトラック爆発

4. WSH 2026 での修正

修正1: パスワードバリデーション

Before
/^(?:[^\P{Letter}&&\P{Number}]*){16,}$/v.test(password)
// ネスト量指定子: ([...]*){16,} → バックトラッキング爆発
After
// 「記号を含むか」を単純にチェック
password.length > 0 && /^[\p{Letter}\p{Number}]*$/v.test(password)
// ↑ 文字・数字のみなら true → 記号なし → エラー
// ネストなし、バックトラッキングなし、O(n)

修正2: 検索クエリの日付パース

Before
const sincePattern = /since:((\d|\d\d|\d\d\d\d-\d\d-\d\d)+)+$/;
//                          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
// (\d|\d\d|\d{4}-\d{2}-\d{2}) が + で繰り返し、さらに外側も +
// 数字文字列に対してバックトラッキング爆発

const slowDateLike = /^(\d+)+-(\d+)+-(\d+)+$/;
//                     ^^^^^  ^^^^^  ^^^^^
// (\d+)+ は典型的な ReDoS パターン
After
// 日付フォーマットを固定長で指定（バックトラッキング不要）
const sincePattern = /since:(\d{4}-\d{2}-\d{2})/;
const dateLike = /^\d{4}-\d{2}-\d{2}$/;
// ↑ 量指定子は固定長 {4}, {2} のみ
// ネストなし、選択肢の重複なし、O(n)

5. 安全な正規表現を書くルール

  1. ネストした量指定子を避ける: (a+)+, (a*)*, (a+){n} は全て危険
  2. 固定長の量指定子を使う: \d{4} は安全。\d+ のネストは危険
  3. 選択肢の重複を排除: (a|ab) → ab? に書き直す
  4. アンカー ($) に頼らない: $ でマッチ失敗するとバックトラッキングが走る
  5. 正規表現より文字列メソッド: str.startsWith(), str.includes() の方が安全

6. 検出ツール

# safe-regex で静的解析
npx safe-regex "/^(a+)+$/"
# → false (unsafe)

# recheck でより詳細な解析
npx recheck "/^(a+)+$/"

# eslint-plugin-regexp
# .eslintrc に追加すると ReDoS パターンを自動検出
{
  "plugins": ["regexp"],
  "rules": {
    "regexp/no-super-linear-backtracking": "error"
  }
}
確認クイズ: /^(a+)+$/ に "aaaaX" を入れると何が起きる？

正規表現エンジンは以下を試みます:

  1. "aaaa" を1つ目の a+ で消費 → "X" で $ にマッチ失敗
  2. "aaa" + "a" の分割を試す → 同様に失敗
  3. "aa" + "aa", "aa" + "a" + "a", "a" + "aaa", ... の全組み合わせ

4文字の "a" を「どの a+ グループに割り当てるか」の全パターンは 2^3 = 8通り。"a" が30文字なら 2^29 ≈ 5億通り。計算量が入力長に対して指数関数的に増加します。

確認クイズ: なぜ /^\d{4}-\d{2}-\d{2}$/ は安全？

全ての量指定子が固定長（{4}, {2}）だからです。固定長量指定子にはバックトラッキングの余地がありません。「4桁の数字」に対して「3桁+1桁に分割して試す」ような分割パターンが存在しないためです。正規表現の安全性を高めるには、可能な限り固定長の量指定子を使うことが有効です。
